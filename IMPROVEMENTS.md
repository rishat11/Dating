# План доработок по соответствию spec.md

**Версия спецификации:** 1.1  
**Дата анализа:** 2025-03-08

Документ описывает разрывы между текущей реализацией и [spec.md](spec.md) и план работ по их устранению.

---

## 1. Язык интерфейса (локаль)

**Спецификация:** п. 3.1, 6, 6.1, 12 — выбор языка русский/английский при первом запуске и в настройках; все тексты бота по `locale` пользователя.

**Текущее состояние:**
- В модели `User` нет поля `locale`.
- Выбор языка при регистрации и в настройках отсутствует.
- Все тексты захардкожены на русском.

**Доработки:**
1. Добавить в `db/models.py` поле `locale: Mapped[str]` (значения `"ru"` / `"en"`), по умолчанию `"ru"`.
2. В онбординг после согласия с правилами (или в начале) добавить шаг выбора языка; сохранять `locale` при создании/обновлении пользователя.
3. Реализовать слой локализации: словари/файлы переводов (например, `i18n/ru.py`, `i18n/en.py`) и функция получения строки по ключу и `user.locale` (или хелпер `t(key, locale)`).
4. Заменить все пользовательские тексты (онбординг, анкета, лента, чаты, уведомления, кнопки) на вызовы локализации с учётом `locale`.
5. Добавить пункт «Настройки» в главное меню; в настройках — смена языка (ru/en) и сохранение в профиле.

## 2. Геолокация и расстояние в ленте

**Спецификация:** п. 3.2, 3.3, 6, 6.1, 7 — геолокация опционально; в ленте показывать ближайших и расстояние до каждого («~2 км»); сортировка по расстоянию.

**Текущее состояние:**
- В модели `User` нет полей `latitude`, `longitude`, `location_updated_at`.
- В анкете и настройках нет приёма/обновления/отключения геолокации.
- В ленте выборка по `func.random()`, без учёта расстояния; расстояние на карточках не отображается.

**Доработки:**
1. Добавить в `User`: `latitude`, `longitude`, `location_updated_at` (все `Optional`/nullable).
2. В анкете (или в настройках) добавить шаг/кнопку «Поделиться местоположением» (Telegram `request_location` или приём `message.location`); сохранять координаты и `location_updated_at`. Реализовать «Отключить геолокацию» (обнуление полей).
3. В сервис ленты (например, `handlers/feed.py` или отдельный `services/feed_service.py`): функция расчёта расстояния Haversine между двумя точками; форматирование приближённого расстояния для отображения («~N км», округление до целых км или диапазон «менее 5 км»).
4. При выборке кандидатов для ленты: если у текущего пользователя есть координаты — сортировать по расстоянию (сначала с координатами, по возрастанию расстояния), затем без координат; для пользователей с координатами добавлять в текст карточки строку с расстоянием.
5. Учитывать в выборке только пользователей с заполненной геолокацией для сортировки «ближайшие»; пользователей без геолокации показывать без расстояния и после ближайших (или вперемешку по политике продукта — зафиксировать в коде/конфиге).

## 3. Очередь и Redis (архитектура)

**Спецификация:** п. 2.1, 2.3 — очередь (Redis Streams / RabbitMQ и т.п.) для расчёта индекса и уведомлений; кэш в Redis для текущего индекса и последних сообщений с TTL и инвалидацией.

**Текущее состояние:**
- Очередь расчёта индекса совместимости реализована in-memory (`asyncio.Queue` в `services/destiny_queue.py`). При рестарте задачи теряются; нет персистентности и масштабирования воркерами.
- Redis в конфиге (`redis_url`) не используется для очереди и кэша.

**Доработки:**
1. При наличии `REDIS_URL`: использовать Redis Streams (или список + BLPOP) для очереди задач расчёта индекса; воркер(ы) забирают задачи из Redis. При отсутствии Redis — сохранить fallback на in-memory очередь (как сейчас).
2. Кэш в Redis: ключ по `match_id` для текущего процента и при необходимости последних N сообщений; TTL по конфигу (например, 1–24 ч); при новом сообщении в чате — инвалидировать кэш этой пары. При расчёте индекса — писать результат в БД и в Redis.
3. Документировать в README или конфиге: для продакшена рекомендуется Redis; без Redis — однопоточный режим без персистентности очереди.

## 4. Проверка статуса матча при расчёте индекса

**Спецификация:** п. 4.6 — при статусе пары «завершён»/«заблокирован» не обновлять индекс.

**Текущее состояние:**
- В `recalc_destiny_index_for_match` и в воркере очереди не проверяется статус матча перед записью нового процента.

**Доработки:**
1. В начале `recalc_destiny_index_for_match` (или в воркере перед вызовом) загружать матч и проверять `match.status == MatchStatus.ACTIVE`. Если не активен — выходить без обновления индекса.

## 5. Пороги разблокировок (Облако воспоминаний)

**Спецификация:** п. 4.3 — уровень 51–70% «Потенциальная пара»: «Облако воспоминаний».

**Текущее состояние:**
- В `services/destiny_service.py` флаг `flag_cloud` выставляется при `new_percent >= 70`.

**Доработки:**
1. Изменить порог для `flag_cloud` на `51` (при `new_percent >= 51`), чтобы разблокировка «Облако воспоминаний» соответствовала диапазону 51–70%.

## 6. Подсказка в чате «до N% осталось N сообщений»

**Спецификация:** п. 4.1 — в шапке/описании чата подсказка вида «+3% за обсуждение музыки | До 50% осталось 1 сообщение».

**Текущее состояние:**
- В шапке чата выводятся прогресс-бар и уровень, подсказки «до N% осталось …» нет.

**Доработки:**
1. По желанию: в сервисе индекса или в отдельной логике оценивать «оставшиеся сообщения до следующего порога» (эвристика по дельте за последние сообщения) и возвращать короткую подсказку.
2. В `_chat_header` или в сообщении при открытии чата добавлять вторую строку с подсказкой (например, «До 50% осталось ~2 сообщения» или причина последнего роста). Можно начать с простой формулы «до следующего уровня осталось X%» без расчёта по сообщениям.

## 7. Настройки и удаление аккаунта (GDPR)

**Спецификация:** п. 3.1 (смена языка в настройках), п. 6.3 — кнопка «Удалить аккаунт», мягкое удаление с периодом восстановления, каскадное удаление/анонимизация.

**Текущее состояние:**
- Нет экрана «Настройки»; в правилах упоминается «удалить аккаунт в настройках», но функционала нет.
- Поле `deleted_at` в `User` есть, логики мягкого удаления и каскада нет.

**Доработки:**
1. Добавить в главное меню кнопку «⚙️ Настройки». В настройках: смена языка (если реализована локаль), включение/отключение геолокации, кнопка «Удалить аккаунт» и ссылка на политику конфиденциальности.
2. Реализовать сценарий «Удалить аккаунт»: подтверждение, установка `deleted_at = now()`, при необходимости уведомление партнёров в активных чатах («собеседник удалил аккаунт»). В выборках (лента, чаты, матчи) фильтровать пользователей с `deleted_at IS NOT NULL`.
3. Опционально: фоновый job или скрипт, который после периода восстановления (например, 30 дней) безвозвратно удаляет или анонимизирует записи с просроченным `deleted_at`.

## 8. Rate limiting и защита от злоупотреблений

**Спецификация:** п. 7.3 — лимиты на запросы по user_id и по чату (сообщений в минуту/час), на ленту и лайки.

**Текущее состояние:**
- Лимит только на лайки в день (`daily_likes_limit`). Нет ограничения сообщений в чате в минуту/час, нет ограничения запросов к ленте.

**Доработки:**
1. Ввести лимиты (конфиг или константы): сообщений в чате в минуту/час на пользователя; запросов «следующая карточка» ленты в минуту.
2. Хранить счётчики в Redis (рекомендуется) или in-memory с TTL; при превышении — сообщение «Слишком много действий, подождите N минут» и не выполнять действие.

---

## 9. Модерация контактов в сообщениях

**Спецификация:** п. 7.1 — автоматическая проверка текста на маркеры обмена контактами (телефон, соцсети, ссылки на личные профили); при обнаружении — предупреждение или заморозка индекса.

**Текущее состояние:**
- Проверки контента сообщений на контакты/ссылки нет.

**Доработки:**
1. Перед сохранением текстового сообщения в чат проверять текст на простые паттерны (телефонные номера, ссылки на vk.com, t.me/username и т.д.). При обнаружении: не отправлять сообщение партнёру (или отправить с подменой), отправить автору предупреждение; при повторении — заморозка индекса по аналогии с негативом.

---

## 10. Логирование и аудит

**Спецификация:** п. 7.4 — логировать матчи, блокировки, жалобы, смену статуса пары; не хранить полный текст сообщений в логах.

**Текущее состояние:**
- Обычное логирование приложения есть; структурированного аудита (события матч/блок/жалоба) нет.

**Доработки:**
1. Ввести отдельный логгер или канал аудита; при создании матча, блокировке, жалобе, завершении чата писать структурированную запись (user_id, match_id, действие, timestamp). В логах не выводить тело сообщений, только id, тип, длину при необходимости.

---

## 11. Хранение и срок жизни данных

**Спецификация:** п. 6.2 — сообщения хранить не более N месяцев (например, 12); геолокация — только актуальная позиция.

**Текущее состояние:**
- Политика удаления старых сообщений не реализована. Геолокация после добавления будет храниться только актуальной — достаточно не вести историю координат.

**Доработки:**
1. Добавить в конфиг параметр `messages_retention_months` (например, 12). Реализовать периодическую задачу (cron/фоновый воркер), которая удаляет или анонимизирует записи в `messages` старше N месяцев; при необходимости перед удалением переносить цитаты в «облако воспоминаний» (если фича реализована).

## 12. Прочее (уже соответствует или мелкие улучшения)

- **Идемпотентность:** обработка по `update_id` в middleware и по `message_id` в очереди расчёта индекса — соответствует спецификации.
- **Webhook:** поддержка webhook в `main.py` по конфигу — есть.
- **Compatibility Index MVP:** лексика, поведение, эмодзи, заморозка при негативе, уведомления о росте и «Спасите индекс» — реализованы.
- **Главное меню:** есть «Анкета», «Лента», «Чаты»; добавить «Настройки» (см. п. 1, 7).
- **Кнопка «Пропустить» в ленте:** реализована как «skip» в callback; текст кнопок можно вынести в локализацию (п. 1).
